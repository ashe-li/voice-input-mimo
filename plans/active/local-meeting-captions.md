## 概要

建立一個獨立的 macOS 會議工具，用於長時間錄音、即時雙字幕、逐字稿保存、會後精修與匯出。整體採 local-first 設計：Swift/SwiftUI 負責產品殼、音訊擷取、字幕 overlay、時間軸、儲存與匯出；外部 Python/MLX sidecar 負責即時 Qwen3-ASR；Rapid-MLX 負責翻譯、摘要、action items 與會議整理。MiMo 先列為 experimental post-process provider，必須通過 benchmark gate 才能進入預設 final path。

既有 `voice-input-mimo` 保留為短語音 dictation 工具。新 app 可以參考它的 ASR/LLM client、overlay 狀態管理、archive retention 與本地服務監控概念，但不應共用同一個 app 流程或 bundle。

MVP 範圍鎖定：mic-only capture、external Qwen sidecar、Rapid-MLX `POST /v1/chat/completions`、Markdown/JSON export。System audio、app-managed sidecar、MiMo final path、SRT/VTT 精準字幕匯出都必須先通過對應 spike 或 benchmark gate。

## 架構說明

### Decision 1: 獨立 Swift App

選擇：建立新的 Swift/SwiftUI/AppKit macOS app。

理由：這是一個 native macOS 會議工具，需要 mic capture、floating subtitles、TCC permissions、長時間背景執行與本地檔案管理。Swift 能直接使用 `AVAudioEngine`、`NSPanel` 與 macOS 權限流程；system audio 之後若加入，才使用 `ScreenCaptureKit`。

排除方案：

| Alternative             | 排除原因                                                                                                                                         |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| 加進 `voice-input-mimo` | 既有 app 是短語音 push-to-talk dictation 與 text injection；meeting lifecycle、transcript storage、長錄音與 overlay 語意都不同，應分成獨立產品。 |
| Electron/JS/TS app      | Web UI 可能較快，但 native audio capture、system audio、overlay、permissions 都會更繞。                                                          |
| Rust primary app        | 適合底層 audio engine，但第一版會增加 bridge/UI 成本，收益不足。                                                                                 |

相容性：既有 `voice-input-mimo` 不變。只復用概念，不復用產品殼。

可擴展性：Swift app 之後可加入 native providers；sidecar 邊界讓 model runtime 可替換。

效能與安全：native audio capture 降低額外開銷；local-first settings 降低資料外洩風險。

### Decision 2: Capture Once, Fan-Out Into Realtime And Recording Lanes

選擇：一個 capture layer 產生 timestamped `AudioFrame` events；兩個獨立 consumers 分別處理 realtime ASR 與 recording。

排除方案：

| Alternative                     | 排除原因                                                        |
| ------------------------------- | --------------------------------------------------------------- |
| 兩套獨立 recorder               | 會造成 timestamp drift、重複 CPU/權限成本，也讓裝置切換更複雜。 |
| ASR pipeline 同時負責 recording | ASR lag 或 crash 會威脅長錄音可靠性。                           |

相容性：MVP 明確只做 mic-only。System audio 只有在 capture graph spike 定義 master clock、source queues、resampling/mixing 與 drift policy 後才能加入。

可擴展性：之後 waveform、diarization、live analytics 都可以再訂閱同一個 bus。

效能與安全：bounded queues 與 recording lane 優先權可避免模型失敗導致 source record 遺失。

### Decision 2b: System Audio Deferred Until Capture Graph Is Proven

選擇：MVP 不實作 system audio。System audio 是 Phase 2+ 能力，必須先完成 capture graph spike。

原因：`AVAudioEngine` microphone capture 與 `ScreenCaptureKit` system audio/microphone outputs 不是同一個簡單來源。實務上會有不同 queue、可能不同 clock domain、不同 sample format 與不同 failure mode。如果不先定義同步與混音策略，後續 timeline、segmentation、export 都可能被迫重做。

必要設計輸出：

| 項目              | 必須定義                                                                          |
| ----------------- | --------------------------------------------------------------------------------- |
| Master clock      | 以哪個 monotonic clock 作為 session-relative ms source。                          |
| Source queues     | mic/system audio 分別進 queue，如何標記 source 與 dropped frames。                |
| Resampling/mixing | 是否保留分軌，何時 mixdown，如何處理 sample rate/channel 差異。                   |
| Drift tolerance   | 允許多少 ms drift；超過時如何校正或標記 transcript uncertainty。                  |
| Timeline policy   | live caption、final ASR、export 使用混音軌、分軌，或 source-specific transcript。 |

排除方案：

| Alternative                         | 排除原因                                                                  |
| ----------------------------------- | ------------------------------------------------------------------------- |
| MVP 直接加 system audio             | 會把 clock sync、TCC、source selection、mixing 全部拉進第一版，風險過高。 |
| 把 mic 和 system audio 直接混成單軌 | 簡化 ASR 但失去 source traceability，且 drift/echo 會變得難以診斷。       |

驗收：沒有 `docs/capture-graph.md` 與手測結果前，不建立 `SystemAudioCapture` implementation。

### Decision 2a: Model-Aware Audio Segmentation

選擇：為 realtime ASR lane、final ASR lane、archival recording lane 設計不同 segmentation policy。

核心原則：app 只 capture 一次，但不強迫所有 downstream consumer 使用同一種 chunk 長度。即時字幕優化 latency 與 partial 穩定度；會後 transcript 優化 context、retryability 與辨識品質。

預設策略：

| Lane                   | 目的                                 | 格式                                       | Segment/window policy                                                       | 備註                                                                       |
| ---------------------- | ------------------------------------ | ------------------------------------------ | --------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| Realtime Qwen lane     | Live captions                        | PCM16 16 kHz mono                          | 100-500 ms frames into 2-6 s rolling windows, 0.5-1.0 s overlap, VAD commit | 針對 partial latency 調整；允許 tail refinement 與 duplicate suppression。 |
| Recording archive lane | Source of truth                      | Native/high-quality PCM or CAF/WAV         | Rotate files every 1-5 min                                                  | 輪替是為 crash safety 與磁碟管理，不是 ASR 品質。                          |
| Final ASR lane         | Post-meeting high-quality transcript | WAV/PCM segment files derived from archive | Speech-aware 30-90 s segments, 2-5 s overlap, hard max cap                  | 針對 batch ASR context 與 retry 調整；保留 source span lineage。           |
| Alignment/export lane  | SRT/VTT and final timeline           | Transcript + timestamps                    | Use live timestamps as coarse anchors; optional forced aligner later        | 避免假裝 MiMo-only text 有精準 word timing。                               |

排除方案：

| Alternative                      | 排除原因                                                                 |
| -------------------------------- | ------------------------------------------------------------------------ |
| 用 realtime chunks 做 final MiMo | 2-6 秒 chunks 對標點/context 太短，會放大邊界 artifacts。                |
| 直接把 1-5 分鐘錄音檔送 MiMo     | 太大，不利 retry、progress 與錯誤隔離，也可能超過實務模型/context 假設。 |
| 單一 universal chunk size        | 既無法最佳化 live latency，也無法最佳化 final transcript 品質。          |

相容性：每個 derived segment 都保存 `sourceRecordingId`、`sourceStartMs`、`sourceEndMs`、`overlapBeforeMs`、`overlapAfterMs`、`policyVersion`、`modelTarget`。這可追蹤 live captions、final ASR output 與 exported subtitles 之間的關係。

可擴展性：不同 provider 可有不同 policy。Qwen3-ASR 1.7B quality mode、WhisperKit、未來 streaming model 都能新增自己的 policy，不需要重寫 capture。

效能與安全：短 live windows 讓 UI 保持 responsive；長 final segments 減少模型 overhead，同時保留可重啟與可重試能力。Raw audio retention settings 控制隱私與磁碟壓力。

### Decision 3: Qwen3-ASR 0.6B-8bit Live，Final ASR Provider Behind Benchmark Gate

選擇：即時字幕使用 Qwen3-ASR 0.6B-8bit via external MLX streaming sidecar。會後 final transcript provider 需經 benchmark gate 決定：候選包含 MiMo-V2.5-ASR batch 與 Qwen3-ASR 1.7B offline reprocess。MiMo 是 experimental provider，不是未驗證主路徑。

排除方案：

| Alternative                        | 排除原因                                                                                |
| ---------------------------------- | --------------------------------------------------------------------------------------- |
| 預設 live 使用 Qwen3-ASR 1.7B-8bit | 品質較好，但 latency 與 memory 成本較高。保留為 quality mode，不作為 MVP default。      |
| MiMo 做 live                       | 目前 MiMo pipeline 是 batch-only；切 chunks 只是應用層 workaround，不是真正 streaming。 |
| 直接把 MiMo 寫死為 final path      | 目前 chunk policy、timestamp 對齊與交付穩定性證據不足；需要與 Qwen 1.7B offline 比較。  |
| Cloud `Qwen3-ASR-Flash-Realtime`   | 即時 API 好用，但使用者需求是 local model，而且它不是開源/本地模型。                    |

相容性：Swift app 透過 provider protocols 與 sidecars 溝通，因此模型選擇可替換，不需重寫 UI/storage。

可擴展性：未來可支援 WhisperKit、FluidAudio/Parakeet、Qwen3-ASR 1.7B quality mode、remote GPU server。

效能與安全：0.6B-8bit 維持 live 低延遲；保留 raw audio 可在不上雲的情況下做高品質 final processing。Final ASR benchmark 必須比較 WER/CER、標點完整度、boundary artifact、latency、memory 與 retry behavior。

### Decision 4: Rapid-MLX 處理 LLM 任務

選擇：MVP 只支援 Rapid-MLX localhost `POST /v1/chat/completions` contract，用於翻譯、摘要、action items 與 transcript cleanup。其他 OpenAI-compatible providers 延後以 adapter 接入。

排除方案：

| Alternative                                | 排除原因                                                                                                 |
| ------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| 在 ASR provider 裡做翻譯                   | 會把 ASR latency 和 LLM latency 綁在一起，backpressure 更難處理。                                        |
| 預設使用 cloud translation/summary         | 與 local-first 期待衝突。                                                                                |
| 一開始支援所有 OpenAI-compatible providers | 各家 chat templates、extra params、`/v1/responses`/`/v1/chat/completions` 差異會讓 MVP contract 變模糊。 |

相容性：既有 `voice-input-mimo` 的 `LLMRefiner` pattern 可作參考，但 meeting app 需要 queueing 與 per-segment persistence。

可擴展性：先把 Rapid-MLX contract 測穩；Ollama、LM Studio、vLLM 之後用 provider adapter 各自處理差異。

效能與安全：LLM queue 有界且可選；API keys 與 transcripts 不應進 log。

### Decision 5: SQLite Plus JSONL Plus Audio Files

選擇：SQLite 儲存 meetings、segments、translations、provider events 與 export metadata；raw audio 分段存在 filesystem；JSONL 記錄 append-only raw events 供 crash/debug。

排除方案：

| Alternative                | 排除原因                                                             |
| -------------------------- | -------------------------------------------------------------------- |
| Pure JSON files            | 長時間 timeline 查詢、migration、export 都較困難。                   |
| SwiftData first            | 樣板較少，但高頻 append 與 schema evolution 的控制不如 SQLite 明確。 |
| 把 audio blobs 存進 SQLite | DB 過大，backup/retention 複雜度增加。                               |

相容性：Swift 可用 SQLite library；sidecar 可透過 filesystem paths 讀取音訊。

可擴展性：支援長會議、搜尋、reprocess jobs、export，不需要把整份 transcript 載進記憶體。

效能與安全：append operations 必須 async；retention settings 必須允許刪除 raw audio。

## 風險與對策

| Risk                                                         | Impact                                                         | Mitigation                                                                                                                    |
| ------------------------------------------------------------ | -------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Mic/system audio clock domain 未定義                         | timeline、segmentation、export 可能被迫重做                    | MVP 鎖 mic-only；system audio 必須先完成 capture graph spike。                                                                |
| ScreenCaptureKit system audio permission/version instability | 系統音訊錄製失敗或讓使用者困惑                                 | 不進 MVP；Phase 2+ 才 feature gate，並加入 diagnostics 與 permission reset docs。                                             |
| Python/MLX sidecar packaging complexity                      | 安裝/更新問題                                                  | MVP 只做 connection monitor、health check、setup diagnostics；不得由 app 安裝、啟動、更新、重啟 sidecar。                     |
| Qwen sidecar transport 未定義                                | RealtimeASRClient 可能假設不存在的 WebSocket/event API         | Spike B 必須先選定單一 transport 與 event envelope；未完成前不得實作 RealtimeASRClient。                                      |
| ASR partial instability/duplicates                           | Live subtitle UX 變差                                          | 使用 finalization states、overlap metadata、text similarity de-dup、partial/final visual styling。                            |
| Wrong chunk size for the target model                        | Live captions 可能 lag，final transcript 可能失去 context/標點 | 為 Qwen live、MiMo final、archive rotation 保留不同 `SegmentationPolicy` profiles；每個 derived segment 記錄 policy version。 |
| Realtime ASR lag/OOM                                         | Captions 延遲或 sidecar crash                                  | Recording lane independent；bounded queues；顯示 degraded mode 與手動重啟指引；app 不直接重啟 sidecar。                       |
| MiMo final provider 未通過 benchmark                         | 會後 final transcript 品質或交付穩定性不足                     | MiMo 只列 experimental；先跑 Spike C，fallback 為 Qwen3-ASR 1.7B offline 或 live transcript。                                 |
| Final transcript lacks precise timestamps                    | SRT/VTT 精準度受限                                             | MVP 只承諾 Markdown/JSON；SRT/VTT 需通過 Spike D timestamp error budget。                                                     |
| 長會議資料量造成磁碟壓力                                     | Storage pressure                                               | Segment audio files、retention policy、per-meeting size display、optional raw audio deletion。                                |
| Sensitive meeting data in logs                               | Privacy breach                                                 | Default transcript logging off；redact provider payloads；API keys 放 Keychain 或安全 local settings。                        |
| Recording legal/consent concerns                             | Product/legal risk                                             | 加入清楚 recording indicator 與文件，提醒使用者遵守會議錄音同意規範。                                                         |

## 驗收標準

- [ ] 有獨立 macOS app skeleton 與自己的 bundle id，且不改動 `VoiceInputMimo` dictation 行為。
- [ ] Mic capture 產生 timestamped audio frames，並由同一 source fan-out 到 realtime 與 recording lanes。
- [ ] Recording lane 可寫入 segmented audio files 與 manifest，至少 60 分鐘不造成 UI freeze。
- [ ] Realtime lane 與 final lane 使用不同 model-aware segmentation policies，且每個 derived ASR segment 都記錄 source span 與 policy metadata。
- [ ] MVP 不建立 `SystemAudioCapture`；system audio 需先有 `docs/capture-graph.md` 與手測結果。
- [ ] Spike B 已選定單一 Qwen sidecar transport 與 event envelope，並以 smoke harness 驗證 sample request/response。
- [ ] MVP app 只做 sidecar connection monitor、health check、setup diagnostics，不負責安裝、啟動、更新、重啟 sidecar。
- [ ] Realtime lane 可連 external local Qwen3-ASR 0.6B-8bit MLX streaming sidecar，並顯示 partial/final captions。
- [ ] Rapid-MLX local provider 可透過 localhost `POST /v1/chat/completions` 從 finalized ASR segments 產生 secondary translation/caption，且不阻塞 recording 或 ASR。
- [ ] Final ASR provider 經 benchmark gate 選定；MiMo 未達標時不得成為 MVP 主路徑。
- [ ] Transcript timeline 持久化到 SQLite，MVP 可匯出 Markdown、JSON；SRT/VTT 需通過 timestamp gate。
- [ ] App 顯示足夠的 permission/provider/queue diagnostics，可用於 debug capture 與 sidecar failures。
- [ ] Security review 確認 logs 無 secrets、預設不記錄 transcript、provider defaults 維持 local-first。
- [ ] Unit/integration tests 覆蓋 chunking、segment merge、provider parsing、storage append、exporters、retry/error behavior。
